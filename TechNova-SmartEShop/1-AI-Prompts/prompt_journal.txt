### PROMPT #1 – Role-Prompting + Requirements Elicitation

**Mục đích:** Bóc tách yêu cầu thô từ khách hàng thành các thành phần BA chuẩn.

**Câu Prompt đã dùng:**

```
Bạn là một Senior Business Analyst (BA) với 10 năm kinh nghiệm trong lĩnh vực 
phân tích hệ thống phần mềm thương mại điện tử. Nhiệm vụ của bạn là tiếp nhận 
đoạn ghi chép yêu cầu thô từ buổi họp với Giám đốc công ty TechNova và chuẩn hóa 
chúng thành tài liệu phân tích chuyên nghiệp.

Dưới đây là bản ghi chép yêu cầu thô:
---
"Tôi muốn một cái hệ thống web để bán hàng. Khách hàng có thể lên đó xem điện 
thoại, laptop, bỏ vào giỏ hàng rồi thanh toán. Cơ mà trước khi mua họ phải đăng 
ký tài khoản và đăng nhập nhé. Hệ thống phải bảo mật, không được để lộ thông tin 
khách.

Về phần nhân viên của tôi, họ cần một trang quản trị (Admin). Trong đó, nhân viên 
kho thì chỉ được thêm, sửa, xóa sản phẩm. Còn quản lý (Manager) thì được xem

truy cập cùng lúc nên API phải chạy nhanh, cơ sở dữ liệu không được sập."

Hãy thực hiện các nhiệm vụ sau:
1. Xác định tất cả các Actors (tác nhân) tham gia hệ thống.
```

### PROMPT #5 – Sinh Entity Classes (Giải quyết N+1 Query)

**Câu Prompt đã dùng:**

```
Bạn là một Senior Java Developer với chuyên môn sâu về Spring Boot và JPA/Hibernate.

Hãy sinh các JPA Entity classes cho hệ thống TechNova Smart E-Shop dựa trên ERD đã thiết kế.
Các Entity cần tạo: User, Role, Product, Order, OrderItem, Cart, CartItem.

Yêu cầu kỹ thuật BẮT BUỘC:
1. Tất cả quan hệ @OneToMany và @ManyToMany PHẢI dùng FetchType.LAZY để tránh N+1 Query
2. Với User có nhiều Role (N-N): Thiết kế @ManyToMany LAZY, bảng trung gian user_roles
3. Với Order có nhiều OrderItem (1-N): Thiết kế @OneToMany LAZY
4. Giải thích rõ tại sao N+1 xảy ra và cách FetchType.LAZY + @EntityGraph giải quyết
5. Dùng Lombok (@Data, @Builder, @NoArgsConstructor, @AllArgsConstructor)
6. Dùng @CreationTimestamp và @UpdateTimestamp cho audit fields
7. Package: com.technova.shop.entity

Trả về code đầy đủ cho từng class, có JavaDoc comment giải thích N+1.
```

### PROMPT #6 – Repository với @EntityGraph

**Câu Prompt đã dùng:**

```
Tiếp tục với vai trò Senior Java Developer, hãy tạo các JPA Repository interfaces 
cho các Entity đã định nghĩa.

Yêu cầu quan trọng – giải quyết N+1 Query:
1. UserRepository: 
   - Phương thức findByUsername(String username) thông thường sẽ KHÔNG load roles 
     (vì LAZY). Khi Spring Security gọi loadUserByUsername() và sau đó truy cập 
     user.getRoles(), JPA sẽ phát sinh thêm N query → đây là N+1.
   - Giải pháp: Tạo @EntityGraph("User.roles") để EAGER fetch chỉ khi cần thiết.
   - Viết method: findWithRolesByUsername(String username) dùng @EntityGraph

2. OrderRepository:
   - findAllByUserId() nếu không dùng EntityGraph sẽ gây N+1 khi load OrderItems
   - Viết method: findAllWithItemsByUserId() dùng @EntityGraph(attributePaths = {"orderItems", "orderItems.product"})

3. CartRepository:
   - findByUserId() cần load CartItems và Product cùng lúc
   - Dùng @EntityGraph cho việc này

Package: com.technova.shop.repository
```

### PROMPT #7 – Cấu hình Spring Security 6 + JWT

**Câu Prompt đã dùng:**

```
Hãy sinh code cấu hình bảo mật đầy đủ cho Spring Boot 3.x / Spring Security 6.

Yêu cầu:
1. JwtTokenProvider:
   - Sử dụng thư viện jjwt (io.jsonwebtoken)
   - Ký bằng HS256, secret key từ application.properties
   - Token có thời hạn sống (expiration) cấu hình được
   - Phương thức: generateToken(), validateToken(), getUsernameFromToken()

2. JwtAuthenticationFilter:
   - Extends OncePerRequestFilter
   - Đọc header Authorization: Bearer <token>
   - Validate token, load UserDetails, set SecurityContext

3. SecurityConfig (Spring Security 6):
   - Cấu hình SecurityFilterChain (KHÔNG dùng WebSecurityConfigurerAdapter đã deprecated)
   - Phân quyền endpoint:
     * /api/v1/public/** → permitAll()
     * /api/v1/auth/** → permitAll()
     * /api/v1/admin/products/** → hasAnyRole("STAFF", "MANAGER")
     * /api/v1/admin/users/** → hasRole("MANAGER")
     * /api/v1/admin/stats/** → hasRole("MANAGER")
     * /api/v1/customer/** → hasAnyRole("CUSTOMER", "MANAGER")
     * Tất cả còn lại → authenticated()
   - Disable CSRF (stateless API)
   - SessionManagement: STATELESS

4. CustomAuthEntryPoint: Trả về HTTP 401 với JSON body:
   {"status": 401, "error": "Unauthorized", "message": "...", "timestamp": "..."}

5. CustomAccessDeniedHandler: Trả về HTTP 403 với JSON body:
   {"status": 403, "error": "Forbidden", "message": "...", "timestamp": "..."}

Package: com.technova.shop.security và com.technova.shop.config
```

### PROMPT #8 – Controller, Service, DTO với Validation

| Kỹ thuật | Prompt # | Mô tả |
